## 1. PROJECT IDENTITY

**Name:** SSC CGL 40-Day Rank-Maker Dashboard ("Conquest")  
**Purpose:** Interactive PWA study planner and mock analytics dashboard for SSC CGL 2026 exam preparation  
**Tech Stack:** Vanilla HTML5 + CSS3 + JavaScript (no frameworks, no build tools)  
**Architecture:** Single Page Application (SPA) with 6 page sections, all client-side  
**Persistence:** Browser localStorage (all state saved client-side)  
**PWA:** Service Worker with cache-first + network-first strategies

## 3. GLOBAL STATE & DATA FLOW

### 3.1 Central State Object (`appState`)
Defined in `js/state.js`. Single source of truth for ALL application state.

```javascript
appState = {
  theme: "dark"|"light",
  currentDay: 1-40,
  examName: string,
  examDate: string (ISO date),
  syllabusProgress: { [subtopicId]: { learned: bool, practiced: bool, mastered: bool } },
  dailyRituals: { drill: bool, vocab: bool, ca: bool, computer: bool },
  sessionActive: bool,
  sessionTime: number (seconds),
  pomoTime: number (seconds),
  pomoMode: "study"|"short-break"|"long-break",
  pomoActive: bool,
  mocks: [{ id, name, date, score, accuracy, rank, breakdown: {quant,reasoning,english,ga}, notes, weakTopicId }],
  notes: [{ id, title, category, subject, content, date }],
  weakAlerts: { [subtopicId]: true }
}
```

### 3.2 Data Flow Architecture
```
JSON Data Files (data/*.json)
    ↓
compile_data.js (Node build tool)
    ↓
js/state.js (inline-compiled constants: SYLLABUS_DATA, PLAN_DATA, FLASHCARDS, EMBEDDED_QUIZZES, GK_STATIC_DATA, TOOLKIT_STATIC_DATA)
    ↓
loadApplicationData() — on DOMContentLoaded
    ↓
loadStateFromStorage() — restores from localStorage
    ↓
appState — single global mutable state object
    ↓
render*() functions — read appState, update DOM
    ↓
saveStateToStorage() — persists to localStorage on every change
```

### 3.3 Inline Compiled Databases
After running `node compile_data.js`, these constants are embedded directly in `js/state.js`:
- `SYLLABUS_DATA` — from `data/syllabus.json`
- `PLAN_DATA` — from `data/plan.json`
- `FLASHCARDS` — from `data/vocab.json`
- `EMBEDDED_QUIZZES` — from `data/quizzes.json` (with aliases: `ENGLISH_QUESTIONS`, `REASONING_QUESTIONS`, `COMP_QUESTIONS`, `GK_QUESTIONS`)
- `GK_STATIC_DATA` — from `data/gk/*.json` (polity, history, geography, science)
- `TOOLKIT_STATIC_DATA` — from `data/toolkit/*.json` (quant, grammar, reasoning, computer, laws, patterns)

### 3.4 Study Content Loading
Separate from the above — study content in `study/` is loaded dynamically via script injection:
- `window.studySubjects` — defined in `study/subjects.js`, loaded via `<script>` tag in HTML
- `window.studyChapters[id]` — loaded on-demand via `loadChapterScript(filePath)` in `js/toolkit.js`
- Each chapter file registers itself: `window.studyChapters = window.studyChapters || {}; window.studyChapters["id"] = "markdown content...";`

## 4. MODULE DEPENDENCY MAP

### 4.1 Import/Export Relationships
All modules use **global scope** (no ES modules, no bundler). Dependencies are implicit via global function calls.

| Module | Calls Into | Called By |
|--------|-----------|-----------|
| **js/state.js** | none (foundation) | ALL modules |
| **js/navigation.js** | state.js (appState, saveStateToStorage), mocks.js, plan.js, speed.js, toolkit.js | app.js (on boot) |
| **js/dashboard.js** | state.js, toolkit.js, mocks.js (renderSvgMockChart), speed.js (parseMarkdown) | app.js (on boot) |
| **js/syllabus.js** | state.js (appState, saveStateToStorage), dashboard.js (renderAll), toolkit.js (tryOpenStudyNote) | navigation.js, dashboard.js |
| **js/plan.js** | state.js (appState, PLAN_DATA, saveStateToStorage), dashboard.js (renderAll, speakText) | navigation.js |
| **js/mocks.js** | state.js (appState, saveStateToStorage), dashboard.js (renderAll, speakText), speed.js (parseMarkdown) | navigation.js |
| **js/toolkit.js** | state.js (appState, saveStateToStorage), speed.js (parseMarkdown) | navigation.js, dashboard.js, mocks.js |
| **js/speed.js** | dashboard.js (speakText), toolkit.js (parseMarkdown used for notes) | navigation.js, dashboard.js |
| **js/app.js** | ALL modules (orchestrator) | index.html |

### 4.2 Cross-Module Coupling Summary
- **Highest coupling:** `js/syllabus.js` ↔ `js/state.js` (syllabus reads/writes appState.syllabusProgress constantly)
- **Second highest:** `js/speed.js` ↔ `js/dashboard.js` (speakText called from 6+ functions in speed.js)
- **Third:** `js/navigation.js` → ALL other modules (routes trigger renders)
- **Lowest coupling:** `js/state.js` (pure foundation, no outgoing deps)
- **Dangerous coupling:** `parseMarkdown()` lives in `js/speed.js` but is used by `js/toolkit.js` and `js/mocks.js` — semantic mismatch

```
User clicks nav item
  → initNavigation handler fires
  → Pauses any active drill (if playing)
  → Hides all .content-page sections
  → Shows target page section
  → Updates global header title + icon
  → Triggers page-specific render:
    - page-syllabus → renderSyllabus()
    - page-plan → renderStudyPlan()
    - page-mocks → renderMockAnalytics()
    - page-toolkit → renderToolkitSubTab() or renderToolkit()
    - page-speed → resetDrillSession() + triggerMathTypesetting()
```

### 9.3 Speed Drill Flow
```
User clicks Start
  → initSpeedDrillsPage → btnDrillPause.onclick
  → drillIsPlaying = true
  → generateDrillQuestion()
    → generateQuestionTextAndAnswer(mode, level)
    → generateMathOptions(correct)
    → Display question + 4 option buttons
    → Start 5-10 second timer
  → User clicks option
    → checkDrillAnswer(chosenVal)
    → Compare with drillAnswerVal
    → Update score/streak/feedback
    → setTimeout(generateDrillQuestion, 1500)
  → On timeout
    → drillAttempts++, drillStreak = 0
    → speakText("Time out")
    → Show correct answer
    → setTimeout(generateDrillQuestion, 1500)
```

### 9.4 Conquest Challenge Flow
```
User clicks "Start Challenge"
  → startChallengeRun()
  → isChallengeActive = true, drillIsPlaying = true
  → challengeTimeRemaining = 900 (15 min)
  → Start interval: decrement timer each second
  → generateChallengeQuestion()
    → Progressive difficulty:
      - Q 1-5: Easy
      - Q 6-15: Medium
      - Q 16-25: Hard
    → generateProceduralMathQuestion(difficulty)
      → Random mode from 11 available
    → Display question + 4 options
  → User clicks option
    → submitChallengeAnswer()
    → If correct: challengeScore++
    → If wrong: speakText("Wrong")
    → After 1500ms: next question
  → End conditions:
    - Q25 completed → endChallengeRun(true)
    - Time runs out → endChallengeRun(false)
    - User aborts → endChallengeRun(false, true)
  → Results: pass if score >= 20/25
```

### 9.5 State Persistence Flow
```
Any state change (checkbox, form submit, day advance, etc.)
  → saveStateToStorage()
  → JSON.stringify(appState)
  → localStorage.setItem("conquestState", ...)
  
On page load:
  → loadStateFromStorage()
  → JSON.parse(localStorage.getItem("conquestState"))
  → Merges into appState (preserving defaults)
```
